title: "Knowledge Management System"
---

Maestro knowledge accumulation is divided into two types: **Constraints** and **Accumulation**. Constraints are coding conventions, architectural decisions, and quality rules—defining "what not to do". Accumulation consists of operational steps, design assets, and debugging experiences—recording "how it was done". The former requires mandatory loading, while the latter requires on-demand retrieval.

```

│   ├── coding-conventions.md       # category: coding

│   ├── review-standards.md         # category: review
│   ├── debug-notes.md              # category: debug
│   ├── test-conventions.md         # category: test
```

## Relationship Between Spec and Knowhow

**Spec is the index and rules, Knowhow is the details and process.** The two are bridged via the `ref` attribute.

| Layer | Positioning | Content Characteristics | Loading Method |
|---|---|---|---|
| Spec (`specs/`) | Index + Rules | Short entries, <200 word summary | Auto-injection (hook) |
| Knowhow (`knowhow/`) | Detailed Documents | Complete steps, code examples | On-demand loading (`wiki load`) |

<details>
<summary>Entry Format Example</summary>

Spec entries use `<spec-entry>` closing tags:

```markdown
<spec-entry category="coding" keywords="auth,token,rotation" date="2026-04-21">

### Token rotation needs email carried through refresh flow

Revoked column must be set rather than deleting tokens.

</spec-entry>
```

Knowhow documents use YAML frontmatter:

```markdown
---
title: OAuth PKCE Authorization Flow
type: recipe
category: coding
keywords: [oauth, pkce, auth]
tool: true
summary: "Use when implementing OAuth 2.0 login for public clients."
---

## Steps
1. Generate code_verifier ...
```

- `category` = **Who is responsible** (determines file routing and agent injection)
- `keywords` = **About what** (cross-category discovery)

## Knowledge Lifecycle

### Stable Identity (sid)

Every `<spec-entry>` is automatically assigned a stable ID (format `S-YYYYMMDD-xxxx`) at creation, used for cross-file references and evolution chain tracking. Legacy entries can be backfilled with `maestro spec backfill-sid`.

### Supersession (Evolution Chain)

When new knowledge replaces old, use supersede to build an evolution chain:

```bash
# 1. Add new entry (sid auto-generated)
maestro spec add coding "New rule" "content" --keywords kw1,kw2 --json
# Output includes sid, e.g. S-20260704-a1b2

# 2. Mark old entry as deprecated
maestro spec supersede <old-sid> --by <new-sid>

# 3. View evolution chain
maestro spec history <sid>
# ○ deprecated  S-20260101-x1y2  "Old rule"
#     ↓
# ● CURRENT     S-20260704-a1b2  "New rule"
```

Superseded entries are automatically set to `status="deprecated"`, excluded from search and agent injection, but still accessible via `--include-deprecated`.

### Dual-Track: Supersede vs Conflict

New knowledge relates to existing entries in two ways — with distinct semantics and operations:

| Relationship | Scenario | Operation | Old entry status |
|-------------|----------|-----------|-----------------|
| **supersede** | New rule replaces old (evolution) | `maestro spec supersede` | `deprecated` (excluded) |
| **conflict** | Both rules plausible (dispute) | `maestro spec conflict mark` | `contested` (de-ranked but retained) |

### Health Check

```bash
maestro spec health
```

Reports: lifecycle statistics (active/deprecated/contested), evolution chain count, dangling/cyclic supersedes detection, overall freshness average.

## Tool — Executable Knowledge

A Tool is a knowhow document marked with `tool: true`, defining **executable business processes**—lightweight workflows, accumulated in the project directory, with self-discovery and self-use characteristics.

```yaml
---
title: Payment Gateway Idempotency Verification
type: recipe
category: test
keywords: [payment, gateway, idempotency]
tool: true
summary: "Use when testing payment endpoints for retry safety."
---

## Steps
1. Generate idempotency key (UUID v4)
2. Submit charge request with key
3. Retry same request — assert identical response
4. Submit different amount with same key — assert 409
5. Verify webhook delivers exactly once
```

`spec load --category test` automatically scans for documents in knowhow/ where `category=test` and `tool=true`, injecting the tool summary along with spec into the agent context.

### Registration and Usage

| Phase | Command | Scenario |
|------|------|------|
| During Planning | `/maestro-tools-register generate` | Standardize business processes |
| After Execution | `/maestro-tools-register extract` | Capture verified operational steps |
| Before Testing | `/maestro-tools-register generate` | Register verification methods for the test agent |
| During Retrospective | `/maestro-tools-register optimize` | Extract reusable processes from artifacts |

Usage: Execute by name `/maestro-tools-execute integration-test`, discover by category `/maestro-tools-execute --category test`, Agent auto-discovery (`spec load` output includes tool summaries).

## Knowledge Graph Integration (KG x Wiki)

When `maestro kg index` generates `knowledge-graph.json`, WikiIndexer automatically indexes KG data as virtual wiki entries:

| KG Data | Wiki Entry | virtualKind | Purpose |
|---------|-----------|-------------|------|
| GraphNode | `kg-{id}` | `kg-node` | Code entities (functions, classes, modules) |
| Layer | `kg-layer-{id}` | `kg-layer` | Architecture layers (CLI, Core, Orchestration) |
| TourStep | `kg-tour-{order}` | `kg-tour-step` | Code tour steps (linked list chain) |

**Edge dual-layer storage**: `related[]` stores top-N related IDs (for wiki graph analysis), `ext.kgEdges[]` stores complete directed heterogeneous edges (for semantic traversal).

**Search degradation**: KG nodes only index title + tags in BM25, avoiding code identifier pollution of regular search.

**Cross-references**: KG nodes automatically match `codebase-comp-*` entries via `filePath`, establishing `ext.semanticDuplicateOf` references.

```bash
# View KG index
maestro wiki list --keyword kg

# Search code entities
maestro wiki search "AuthMiddleware"

# Code change impact analysis
maestro kg diff-wiki

# KG node details (with related wiki entries)
maestro kg explain <node-id>
```

## Domain Knowledge System

The Domain system manages the project domain glossary, providing domain context for spec injection and code comprehension. Core modules include `domain-loader.ts` (term CRUD + file locking), `domain-scanner.ts` (codebase scanning for candidate terms), and `domain-matcher.ts` (CJK-aware term matching).

### CLI Subcommands

| Subcommand | Responsibility |
|--------|------|
| `domain init` | Initialize `.workflow/domain/` with empty `glossary.json` |
| `domain add <canonical> <definition>` | Add a domain term (supports aliases, keywords, relationships, tier) |
| `domain list` | List all terms, supports `--status active\|deprecated` filtering |
| `domain show <id>` | Show term details (including concept_ref document content) |
| `domain update <id>` | Update term (definition, aliases, relationships, keywords, tier) |
| `domain remove <id>` | Remove term (checks reference dependencies, returns warnings) |
| `domain search <query>` | Search terms (canonical + aliases + definition + keywords) |
| `domain discover` | Scan codebase for candidate terms (based on interface/type/enum/class/route/doc) |
| `domain import` | Import terms from external sources (`--from context-package \| @<file>`) |
| `domain deprecate <id>` | Soft-delete term (mark deprecated, optionally specify successor) |
| `domain validate` | Validate `glossary.json` schema and relationship integrity |

### Term Structure

Each term contains: `id` (kebab-case), `canonical` (display name), `definition`, `aliases[]`, `keywords[]`, `relationships[]`, `tier` (core/extended/peripheral), `status` (active/deprecated), `source` (manual/discover/import), and optional `concept_ref` (path to detailed concept document).

### Integration with Spec Injection

`spec-injector` and `keyword-spec-injector` match domain terms in the prompt via `domain-matcher` before injecting spec entries. Matched term definitions are injected as context prefix, helping the agent understand project-specific concepts.

## Credibility Assessment System

`credibility.ts` implements **exponential decay** based knowledge credibility scoring, while `spec-analytics.ts` records injection logs for improvement analysis.

### Decay Model

```
factor = floor + (1 - floor) * e^(-λ * age_days)
λ = ln(2) / half_life
```

| Node Type | Half-Life (days) | Description |
|----------|-------------|------|
| domain | 180 | Domain terms change slowly |
| spec | 60 | Constraint rules at moderate update frequency |
| knowhow | 30 | Operational knowledge decays faster |
| issue | 14 | Issue status changes frequently |
| project/roadmap/note | 90 | General moderate decay |

- `floor = 0.3`: Minimum credibility floor
- `ceiling = 1.2`: Search hits can boost up to this limit
- `warningThreshold = 0.5`: Below this triggers a low credibility warning

### Storage and Updates

`CredibilityStore` uses a SQLite `credibility` table, recording `search_hits`, `consumption_count`, `last_hit_at`, `last_consumed_at`, and `content_changed_at`. When content changes, the `content_hash` comparison resets the decay starting point. Supports `incrementSearchHits` (batch) and `incrementConsumption` (single) for usage tracking.

### Spec Analytics

`spec-analytics.ts` records three log types to `.workflow/spec-analytics.jsonl`:

| Type | Source | Recorded Content |
|------|------|---------|
| `injection` | spec-injector / keyword-spec-injector / spec-injection-plugin | Agent type, matched categories, injected entry count, budget action, matched keywords |
| `cli` | CLI endpoints | Command name, arguments |
| `hook` | Workflow hooks | Hook name, duration, outcome |

Aggregated statistics provide: injection hit rate by source/agent type/category, keyword Top-N ranking, CLI usage frequency, and hook invocation stats. Log files auto-rotate (default 5MB).

## KG Custom Extractor Plugin Mechanism

`plugin-engine.ts` supports two plugin modes to extend knowledge graph code extraction capabilities, configured via `.workflow/kg/extractors.yaml`.

### Two Plugin Modes

| Mode | Configuration | Execution | Use Case |
|------|---------|---------|---------|
| **declarative** | Define `rules[]` in YAML | Regex/call/assignment pattern matching | Simple symbol extraction (constants, routes, decorators) |
| **script** | `.workflow/kg/extractors/*.mjs` | Dynamic `import()` + `extract(ctx)` call | Complex logic (AST traversal, cross-file analysis) |

### Declarative Rule Types

- `regex`: Regex matching, supports `$1`-`$9` template name extraction
- `call`: Function call pattern (e.g., `builder.define_constant($NAME, $_)`)
- `assignment`: Assignment pattern, supports `module`/`class`/`any` scope filtering

### Script Plugin API

Script plugins export an `extract(ctx)` function. `PluginContext` provides:

- `ctx.filePath` / `ctx.sourceCode` / `ctx.language`: File information
- `ctx.findAll(nodeType)`: Walk tree-sitter AST to find nodes of given type
- `ctx.text(startLine, endLine)`: Extract source lines
- `ctx.makeSymbol(input)`: Build standardized symbol object

### Merge Strategy

Plugin extraction results are merged with core tree-sitter results. The conflict policy is controlled by `defaults.conflictPolicy`: `merge-metadata` (default, keep core symbol), `plugin-wins` (plugin overrides), `core-wins` (core retains).

## CooldownGuard Abstraction

`cooldown-guard.ts` provides cross-process cooldown guards, implementing throttling across subprocess invocations via tmpdir bridge files.

### Core API

```typescript
class CooldownGuard {
  shouldRun(sessionId): boolean   // Whether within cooldown window
  markDone(sessionId, extra?)     // Mark completion, write timestamp
  timeSinceLastMs(sessionId)      // Milliseconds since last trigger
}
```

### Pre-configured Instances

| Instance | Cooldown | Purpose |
|------|---------|------|
| `kgSyncGuard` | 30 seconds | KG sync throttling, avoids frequent index rebuilds |
| `kgInitGuard` | 5 minutes | KG initialization throttling, avoids repeated full scans |

Bridge files are stored in system tmpdir (`maestro-kg-sync-{sessionId}.json` / `maestro-kg-init-{sessionId}.json`), containing a `last_trigger` timestamp. `shouldRun()` compares the current time with the last trigger time and returns `true` if the cooldown window has elapsed.

## Script Plugins Security Policy

Script plugins (`.mjs` files) are **disabled by default** and must be explicitly enabled to prevent execution of untrusted code.

### Enabling

```bash
# Explicit CLI enable
maestro kg sync --allow-extractor-scripts

# Passed through the code-extractor call chain
codeExtractor.extract({ allowScripts: true })
```

### Security Behavior

| Scenario | Behavior |
|------|------|
| `.mjs` files exist but not enabled | Outputs stderr warning, skips all script plugins |
| Enabled | Dynamic `import()` loading; on `extract()` failure, handled per `onError` policy |
| Plugin lacks `export function extract` | Silently skipped |
| Declarative plugins | Always loaded, not subject to this security policy |

`warn` (default): Plugin failure outputs warning and continues execution; `fail`: Plugin failure throws error and terminates extraction.

## Knowledge Flow Panorama

```
Execution Artifacts         Extraction                Storage                 Consumption
─────────                  ─────                    ─────                  ─────
Analysis Session ─────┐                              ┌─→ specs/     ─→ spec-injector → agent
Debug Records ────────┼──→ /manage-harvest ──────────┼─→ knowhow/   ─→ wiki load → on-demand
Plan Documents ───────┤    /quality-retrospective    ├─→ issues/    ─→ manage-issue → track
Code Changes ─────────┘    /learn-decompose          └─→ learnings  ─→ keyword-injector → context
```

Progressive Fill——Automatic accumulation at each stage:

```bash
maestro-init    → spec-setup (skeleton + scan)
maestro-analyze → lock decisions → arch, code patterns → coding
maestro-plan    → design conventions → coding/arch, test strategy → test
maestro-execute → lessons learned → learning, root causes → debug
maestro-execute → built-in verification (E2.7) → quality discoveries → review
```
