# .workflow/ Artifact Directory System

---

## 1. Global Overview

### Design Philosophy

`.workflow/` is Maestro's **Single Workspace**, storing all workflow state and artifacts for a project. It follows three core principles:

1. **Single Workspace** — All commands read from and write to the same `.workflow/` directory, with centralized state management
2. **Artifact Registry** — Every command output is registered through `state.json.artifacts[]`, enabling cross-phase tracking
3. **Phase-Driven** — Artifacts flow through the analyze → plan → execute → verify lifecycle

### Core Principles

- Each directory/file has a single responsibility with no overlapping storage
- `.workflow/` is added to `.gitignore` and excluded from version control (except `project.md` as an optional exception)
- All artifact paths are relative to the `.workflow/` root directory

### Scopes

| Path | Scope | Description |
|------|-------|-------------|
| `.workflow/` | Project-level | All workflow state for the current project |
| `~/.maestro/` | Global-level | Cross-project templates, configs, overlays |
| `.workflow/collab/` | Team-level | Human team collaboration, strictly isolated from `.workflow/.team/` (Agent bus) |

## 3. Core File Details

### 3.1 State Management

#### state.json — Project State Machine and Artifact Registry

`state.json` is the core of `.workflow/`, recording project state and all registered artifacts.

```json
{
  "version": "3.0",
  "status": "idle|active",
  "project_name": "project-name",
  "current_milestone": "M1",
  "current_phase": 1,
  "last_release_version": "0.1.0",
  "last_release_at": "ISO-8601",
  "codebase_last_rebuilt": "ISO-8601",
  "codebase_last_refreshed": "ISO-8601",
  "milestones": [
    {
      "id": "M1",
      "name": "milestone-name",
      "status": "active|completed|forked",
      "phases": [1, 2, 3]
    }
  ],
  "milestone_history": [
    {
      "milestone_id": "M1",
      "completed_at": "ISO-8601",
      "artifact_count": 5
    }
  ],
  "artifacts": [
    {
      "id": "ANL-001",
      "type": "analyze",
      "milestone": "M1",
      "phase": 1,
      "scope": "phase",
      "path": "scratch/20260513-analyze-P1-auth",
      "status": "completed",
      "depends_on": null,
      "harvested": false,
      "created_at": "ISO-8601",
      "completed_at": "ISO-8601"
    }
  ]
}
```

**Key Field Descriptions**:

| Field | Description |
|-------|-------------|
| `version` | Schema version, used for `maestro-update` migration |
| `current_milestone` | Current active milestone ID |
| `current_phase` | Current active phase number |
| `artifacts[]` | Artifact registry, all command outputs are registered here |
| `milestones[]` | Milestone list, `status` can be `active`/`completed`/`forked` |
| `milestone_history[]` | Archive records for completed milestones |

#### config.json — User Configuration

```json
{
  "granularity": "phase|milestone|task",
  "workflow_agents": true,
  "gate_preferences": {
    "review_level": "standard",
    "test_coverage": 80
  }
}
```

Created by `maestro-init`, recording the workflow preferences selected by the user during initialization.

#### project.md — Project Definition

Created by `maestro-init`, containing:

- **Core Value** — Project core value proposition
- **Requirements** — Requirements list (Validated / Active / Out of Scope)
- **Key Decisions** — Key decision records
- **Context** — Context summary updated after milestone completion

### 3.2 Planning and Roadmap

#### roadmap.md — Milestone/Phase Structure

Created by `maestro-roadmap` (generated in both light and full modes). Contains:

- Milestone list with phase breakdown
- Success criteria, dependencies, and requirements mapping for each phase
- Phase Progress Table

#### scratch/ — Execution Artifact Directory

All command outputs are written to subdirectories under `scratch/`.

**Naming Convention**: `{YYYYMMDD}-{type}[-P{N}|-M{N}]-{slug}`

| Part | Description | Example |
|------|-------------|---------|
| `{YYYYMMDD}` | Creation date | `20260513` |
| `{type}` | Artifact type | `analyze`, `plan`, `review`, `debug`, `test`, etc. |
| `P{N}` | Phase scope prefix | `P1`, `P2` (phase-level artifacts) |
| `M{N}` | Milestone scope prefix | `M1` (milestone-level verification) |
| `{slug}` | Content summary | `auth-system` |

Sorted by date first for easy chronological ordering. Scope prefixes enable quick identification at the directory level.

**Artifact Subdirectories by Type**:

| Artifact Type | Directory Pattern | Primary Files | Generating Command |
|---------------|-------------------|---------------|-------------------|
| Analysis | `*analyze*` | discussion.md, analysis.md, conclusions.json, context.md | maestro-analyze |
| Plan | `*plan*` | plan.json, .task/TASK-*.json | maestro-plan |
| Execute | (inside plan directory) | .summaries/TASK-{NNN}-summary.md | maestro-execute |
| Verify | `*verify*` | verification.json | maestro-verify |
| Review | `*review*` | review.json | quality-review |
| Debug | `*debug*` | understanding.md, evidence.ndjson | quality-debug |
| Test | `*test*` | uat.md, test-results.json, coverage-report.json | quality-test |
| Auto-test | `*auto-test*` | report.json, scenarios.csv, reflection-log.md | quality-auto-test |
| Brainstorm | `*brainstorm*` | guidance-specification.md, .brainstorming/ | maestro-brainstorm |
| Collab | `*collab*` | collab-report.md, context.md, conclusions.json | maestro-collab |
| UI design | `*ui-design*` | MASTER.md, design-tokens.json, animation-tokens.json | maestro-ui-craft --chain build |
| Retrospective | (writes to retrospective.json/md + specs/learnings.md) | | quality-retrospective |

### 3.3 Knowledge System

#### specs/ — 6 Types of Specification Files

Specification files use the `<spec-entry>` closed-tag format and are automatically indexed to `wiki-index.json` by WikiIndexer.

| File | Category | Core File | Description |
|------|----------|-----------|-------------|
| `coding-conventions.md` | coding | Yes | Coding conventions, always created |
| `architecture-constraints.md` | arch | Yes | Architecture constraints, always created |
| `knowhow.md` | — | Yes | Knowledge index, always created |
| `quality-rules.md` | review | No | Quality rules, created when linter/CI detected |
| `test-conventions.md` | test | No | Test conventions, created when test framework detected |
| `debug-notes.md` | debug | No | Debug notes, created on demand |
| `review-standards.md` | review | No | Review standards, created on demand |
| `learnings.md` | learning | No | Learning records, cross-phase queries |

**Spec Scope and Directory Mapping**:

| Scope | Directory | ID Prefix |
|-------|-----------|-----------|
| project | `.workflow/specs/` | `spec:project:` |
| global | `~/.maestro/specs/` | `spec:global:` |
| team | `.workflow/collab/specs/` | `spec:team:` |
| personal | `.workflow/collab/specs/{uid}/` | `spec:personal:{uid}:` |

**`<spec-entry>` Format**:

```xml
<spec-entry category="coding" keywords="exports,naming" date="2026-05-13" source="spec-add" roles="implement">
  Specification content...
</spec-entry>
```

#### knowhow/ — 9 Knowledge Document Prefixes

All knowledge documents are stored in the `.workflow/knowhow/` directory with the filename format `{PREFIX}-{YYYYMMDD}-{HHMM}.md`.

| Prefix | Type | Token | Description |
|--------|------|-------|-------------|
| KNW- | session (compact) | compact/session/compress/save | Session state compression |
| TIP- | tip | tip/note/record/quick | Quick tips |
| TPL- | template | template/tpl/template | Code/config templates |
| RCP- | recipe | recipe/rcp/recipe/steps | Step-by-step guides |
| REF- | reference | reference/ref/reference | External document summaries |
| DCS- | decision | decision/dcs/decision/adr | Architecture decision records (proposed/accepted/superseded) |
| AST- | asset | asset/asset/asset/contract | Reusable assets (api-contract/data-model/prompt/config) |
| BLP- | blueprint | blueprint/blp/blueprint | Architecture blueprints |
| DOC- | document | document/doc/document | General documents (fallback) |

Each document type uses YAML frontmatter containing `title`, `type`, `category`, `created`, `tags`, `source`, and other fields. Tool-type documents (`tool: true`) can be auto-discovered by `spec load` and consumed by downstream agents.

#### wiki-index.json — Unified Index

Automatically generated and maintained by `WikiIndexer`, indexing the following content sources:

- **Single files**: `project.md` (type=project), `roadmap.md` (type=roadmap)
- **specs/**: All `.md` files in all scope directories (type=spec), including child `<spec-entry>` nodes
- **knowhow/**: All `.md` files (type=knowhow), category inferred from prefix
- **issues/**: Entries in `issues.jsonl` (type=issue)
- **scratch/ roadmap**: Single-file parsing

The index uses a lightweight format (stripped body/raw/ext) for BM25 search, backlinks, health scores, etc.

#### learning/ — Learning Data

Learning-related data is stored in two locations:

- `specs/learnings.md` — Learning insights and tips stored as `<spec-entry>` blocks
- `knowhow/.maestro-learn/` — `maestro-learn` session state directory

### 3.5 Milestone Management

#### milestones/{M}/ — Archive Structure

Archives are created by `maestro-milestone-complete` when a milestone is completed:

```
milestones/
└── M1/
    ├── artifacts/            # Archived artifacts moved from scratch/
    │   └── (complete migration of original scratch subdirectory)
    ├── audit-report.md       # Milestone audit report (PASS/FAIL)
    ├── summary.md            # Milestone summary
    └── roadmap-snapshot.md   # Roadmap snapshot
```

**Archive Process**:
1. `maestro-milestone-audit` validates completeness → generates `audit-report.md`
2. `maestro-milestone-complete` moves scratch artifacts into `milestones/{M}/artifacts/`
3. Entries in `state.json.artifacts[]` are moved to `milestone_history[]`
4. Final knowhow is extracted and the workflow advances to the next milestone

### 3.6 Collaboration and Sessions

#### collab/ — Human Team Collaboration

`.workflow/collab/` is the human team collaboration space, **strictly isolated from `.workflow/.team/` (Agent message bus)**.

```
collab/
├── specs/                  # Team-level specs
└── specs/{uid}/            # Personal-level specs (isolated by user ID)
```

WikiIndexer scans `collab/specs/` and its subdirectories, creating index entries for team and personal specs.

#### .maestro/ — Session State

Stores runtime session state for various agents. Users generally do not need to interact with this directly.

```
.maestro/
├── maestro-{YYYYMMDD-HHmmss}/
│   └── status.json         # maestro main coordinator session
├── ralph-{YYYYMMDD-HHmmss}/
│   └── status.json         # ralph adaptive engine session
├── player-{YYYYMMDD-HHmmss}/
│   └── status.json         # player template playback session
└── coord-{timestamp}/      # Coordinator session
    └── walker-state.json   # Walker state
```

**Session Schema Differences**:

| Session | source Field | Key Fields |
|---------|-------------|------------|
| maestro | `"maestro"` | intent, task_type, chain_name, steps[], waves[] |
| ralph | `"ralph"` | lifecycle_position, passed_gates[], context{scratch_dir, plan_dir, ...} |
| player | — | template_id, template_path, last_checkpoint |

#### .team/ — Agent Team Message Bus

`.workflow/.team/{session-id}/.msg/messages.jsonl` is the persistent log for Agent team communication. Managed by the `team_msg` tool from `maestro-tools`, supporting `log`, `broadcast`, `read_mailbox`, and other operations.

## 4. Artifact Registry

### artifacts[] Complete Schema

```json
{
  "id": "ANL-001",
  "type": "analyze",
  "milestone": "M1",
  "phase": 1,
  "scope": "phase",
  "path": "scratch/20260513-analyze-P1-auth",
  "status": "completed",
  "depends_on": null,
  "harvested": false,
  "created_at": "2026-05-13T10:00:00Z",
  "completed_at": "2026-05-13T11:00:00Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Artifact ID, format `{TYPE_PREFIX}-{NNN}` |
| `type` | string | Artifact type identifier |
| `milestone` | string | Owning milestone |
| `phase` | number/null | Owning phase number (null for cross-phase/standalone) |
| `scope` | string | Scope: `phase`, `milestone`, `adhoc`, `standalone` |
| `path` | string | Path relative to `.workflow/` |
| `status` | string | `created` / `completed` / `failed` |
| `depends_on` | string/null | Upstream artifact ID this depends on |
| `harvested` | boolean | Whether knowledge has been extracted by harvest |
| `created_at` | string | Creation time (ISO-8601) |
| `completed_at` | string/null | Completion time (ISO-8601) |

### Type ID Naming Convention

| Type | ID Prefix | Generating Command |
|------|-----------|-------------------|
| analyze | ANL-{NNN} | maestro-analyze |
| plan | PLN-{NNN} | maestro-plan |
| execute | EXC-{NNN} | maestro-execute |
| verify | VRF-{NNN} | maestro-verify |
| review | REV-{NNN} | quality-review |
| debug | DBG-{NNN} | quality-debug |
| test | TST-{NNN} | quality-test |
| brainstorm | BRN-{NNN} | maestro-brainstorm |
| collab | CLB-{NNN} | maestro-collab |
| brainstorm (alias) | BSY-{NNN} | maestro-brainstorm (session) |
| wiki-digest | WBR-{NNN} | wiki-digest |

Numbering is auto-incremented by `nextArtifactId(artifacts, type)`.

### Artifact Lifecycle

```
created → completed → harvested → archived
                     ↘ failed
```

| Stage | Trigger | Action |
|-------|---------|--------|
| created | Command starts execution | Write to artifacts[], status="created" |
| completed | Command finishes successfully | Update status="completed", record completed_at |
| failed | Command execution fails | Update status="failed" |
| harvested | manage-harvest extracts knowledge | Update harvested=true |
| archived | maestro-milestone-complete | Move to milestone_history[], scratch moved to milestones/{M}/artifacts/ |

## 6. Naming Convention Quick Reference

### Scratch Directory Naming

**Format**: `{YYYYMMDD}-{type}[-P{N}|-M{N}]-{slug}`

| Component | Possible Values | Description |
|-----------|----------------|-------------|
| `{YYYYMMDD}` | Date | Creation date, ensures chronological sorting |
| `{type}` | analyze, plan, verify, review, debug, test, auto-test, brainstorm, collab, ui-design | Artifact type |
| `P{N}` | P1, P2, ... | Phase scope (phase-level artifacts) |
| `M{N}` | M1, M2, ... | Milestone scope (milestone-level verification) |
| `{slug}` | kebab-case | Content description summary |

**Special Rules**:
- `adhoc`/`standalone` artifacts have no scope prefix
- `execute` artifacts are generated inside the plan directory (`.summaries/`), not as standalone directories

### Knowhow File Prefixes

| Prefix | Type | Category |
|--------|------|----------|
| KNW- | session | session |
| TIP- | tip | tip |
| TPL- | template | template |
| RCP- | recipe | recipe |
| REF- | reference | reference |
| DCS- | decision | decision |
| AST- | asset | asset |
| BLP- | blueprint | blueprint |
| DOC- | document | document |

**Filename Format**: `{PREFIX}-{YYYYMMDD}-{HHMM}.md` (e.g., `KNW-20260513-1430.md`)

**Special Prefixes** (learn command outputs):
- `KNW-follow-{slug}-{date}.md` — learn-follow
- `KNW-decompose-{slug}-{date}.md` — learn-decompose
- `KNW-retro-{date}.md` / `.json` — learn-retro
- `KNW-opinion-{slug}-{date}.md` — learn-second-opinion
- `KNW-investigate-{slug}/` — learn-investigate (directory form)
- `KNW-digest-{slug}-{date}.md` — wiki-digest
